# AI Agent Operating Protocol

This document defines the mandatory operating constraints for work in the `tableau-ai-pr-agent` repository.

## 1. Repository Check

- Always confirm the active repository before making changes.
- Required pre-flight checks:
  - current working directory
  - `git remote -v`
  - `git status`
  - `package.json` `name`
  - README title
- Expected repository:
  - `Chasoso/tableau-ai-pr-agent`
  - app name: `tableau-ai-pr-agent`
- If the repository points at `tableau-chat-extension` or any other old repo, stop and report it. Do not edit files.

## 2. Path and File Handling

- Use paths relative to the repository root.
- Before reading a file, verify that the path exists and is the expected file or directory.
- Use small, targeted reads and edits.
- Prefer `rg` / `rg --files` for search.
- Use `apply_patch` for manual file edits.
- Do not use destructive commands such as `git reset --hard` or `git checkout --` unless explicitly requested.

## 3. Scope Control

- Keep each request to one phase or one focused task unless the user explicitly asks for more.
- Do not change frontend, backend, infra, or GitHub Actions outside the requested scope.
- Do not remove chat / Notion related code unless the user explicitly asks for removal or the task is a safe cleanup step.
- If a change could affect deployment, infrastructure, or user-facing behavior broadly, raise it first instead of silently applying it.

## 4. Secrets and Sensitive Data

- Never commit `.env` files, tokens, webhook URLs, AWS keys, Tableau secrets, Cognito secrets, or Google OAuth secrets.
- Do not print secrets, AWS account IDs, JWTs, or webhook URLs in docs or logs.
- Prefer placeholders and descriptive labels over sample secrets.
- Treat log output as potentially user-visible and redact sensitive values.

## 5. Validation

- After code changes, run the smallest relevant validation first.
- Prefer one or more of the following as appropriate:
  - `npm run lint`
  - `npm run typecheck`
  - `npm run test:unit`
  - `npm run test:e2e`
  - `npm run build`
- If the change touches UI behavior, verify the relevant frontend tests.
- If the change touches backend logic, verify backend unit or integration tests.
- If the change touches both, run the combined root checks when practical.

## 6. Architecture Boundaries

- Keep business logic in backend services.
- Keep the frontend focused on UI, form state, and API calls.
- Keep Tableau / MCP / Slack / S3 / auth integration logic server-side where possible.
- Preserve the separation between preview-only UI and side-effecting operations.

## 7. Legacy Code and Migration Rules

- This repository is a migration from the older chat-based extension.
- Legacy chat / Notion code may remain for compatibility, but do not remove it casually.
- When renaming code, prefer `action-run`, `pr-agent`, `draft`, `campaign`, or similar domain names where the change is safe.
- If a rename or delete could break compatibility, schedule it as a separate follow-up task.

## 8. Operational Expectations

- Keep docs aligned with the current code and env names.
- Prefer explicit runbooks for setup, demo, troubleshooting, and teardown.
- Keep demo mode and fallback behavior clearly documented.
- Keep traceability in logs through run IDs or equivalent correlation IDs.

## 9. Change Reporting

- After completing a task, report:
  - changed files
  - commands run
  - success or failure
  - any unfinished work
  - the next recommended step

## 10. Safety Defaults

- Default to non-destructive changes.
- Ask before doing any change that could delete data, remove infrastructure, or alter external integrations.
- If the repo state is unclear, verify before editing instead of guessing.
